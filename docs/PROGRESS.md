# 📋 CoverPicker 项目进度日志

## 🚨 核心开发规则

1.提供完整代码：任何时候，需要提供修改好的完整代码（可以适当精简），以便用户直接覆盖。不允许仅给出片段代码、部分方法，让用户自行替换。输出完整代码时，不需要告诉用户"我只修改了xxx，但是为了完整，我还是提供xx"。如果代码没有变化，则不需要重新提供完整代码。

2.确认现象与需求：除非用户仅给出报错日志，其他时候必须向用户确认错误或问题的现象，以及用户的需求，用户同意后再开始修改代码。当用户提供了截图来说明问题时，如果读不懂一定要如实告知用户，并询问详细的错误，严禁自己胡乱解读、胡乱修改。当修改方案与用户需求完全一致时，不需要请用户确认。只有当方案与用户需求不同、或存在多种可选方案时，才需要向用户确认。严禁用户描述完现象，未经确认直接输出代码。

3.禁止擅自删改功能：任何时候，不允许未经用户授权，删除或修改coverpicker已有的功能、布局、按钮等。如果确实需要更改，必须取得用户确认。如果不确定用户原来的代码已实现什么功能，绝不允许擅自开始修改或生成简化的文件代码，务必向用户询问、索取原始代码。如有必要，可以增加调试日志，不需要请用户确认。每次调试，用户都会使用你最后提供的版本完整覆盖，不需要对此提出质疑，不需要就此向用户确认。

4.自查后输出：输出代码前必须进行三轮独立的审查：

第一轮（技术审查）：检查代码是否有语法错误、逻辑漏洞、变量名错误、缩进错误、导入缺失等技术性问题。

第二轮（需求审查）：以用户的视角模拟运行，在脑海中逐行执行代码，检查修改后的代码是否真正实现了用户描述的需求，而不是仅仅"能运行"。

第三轮（规则审查）：仔细核对前面3条规则，是否有违反任何一条。

三轮审查全部结束后，必须用中文输出一份简短的"需求实现确认清单"，至少包含，确认修改后的代码确实实现了用户的需求，且未违反核心规则。不允许输出"已自查3次，确认无误"这样空洞的结论，必须具体说明需求如何被实现。

5.简洁输出原则：当用户已明确确认修复方向后，回答必须直接给出最终结果，禁止包含以下冗余内容：

对用户已确认方案的重复阐述

对问题原因的事后分析（只应在初次诊断时说明）

多个方案对比分析

冗长的实现步骤描述

对之前已提供过的代码的重复引用

正确格式：用户确认后，只输出"好的，我来实现："+ 完整代码文件。如果需要解释，不超过3句话。

## 🔴 已修复问题（v2.5.2）

### 问题4：长视频路径显示不完整 —— ✅ 已修复

**现象**：特定长文件名视频的路径在左侧信息区显示不完整，末尾被截断；在不同缩放比例下表现不一致。

**根本原因**：
1. `QLabel` 的 `wordWrap` 在包含反斜杠 `\` 的长字符串中无法正确计算所需总高度
2. 即使给予足够空间，`QLabel` 也不一定显示所有行（Qt/QLabel 的已知行为限制）
3. 信息组的垂直拉伸因子为0，无法随窗口扩展

**修复方案**：
1. 将 `info_path` 从 `QLabel` 替换为 `QTextEdit`，配置为只读、无边框、背景透明
2. `QTextEdit` 使用完整的文档引擎，能正确计算每行文本高度并自动调整
3. 信息组拉伸因子从0改为1，使其能随窗口垂直扩展


### 问题8：视频状态图标优先级错误 —— ✅ 已修复

**现象**：视频既有收藏（⭐）又有导出（✅）时，列表显示⭐而非✅。

**根本原因**：`get_video_state_icon` 中优先级顺序为 ⭐ > ✅ > 👁️，与需求不符。

**修复方案**：将优先级改为 ✅ > ⭐ > 👁️（符合 PRODUCT.md 7.1）。


### 问题9：NAS收藏静默回退到临时目录 —— ✅ 已修复

**现象**：NAS路径不可写时，收藏截图静默保存到临时目录，用户无感知。

**根本原因**：`_save_favorite_to_nas` 方法在保存失败时直接返回临时路径，未向用户提示。

**修复方案**：
1. `_save_favorite_to_nas` 返回值从 `str` 改为 `Tuple[str, bool]`，返回路径和是否成功保存到NAS
2. `favorite_selected` 中记录失败次数，通过 `_notify_progress` 向用户显示提示


### 新增修复：细选窗口导出后视频列表图标不刷新 —— ✅ 已修复

**现象**：从收藏夹进入细选，导出截图后，视频列表状态图标仍为⭐，未变为✅。

**根本原因**：`zoom_dialog.py` 的 `export_selected` 方法导出成功后未调用 `_notify_data_changed()` 触发主界面刷新。

**修复方案**：在 `export_selected` 中导出成功后添加 `self.controller._notify_data_changed()`。

## 📦 自动备份与恢复功能说明

### 备份内容

自动备份功能保存的当前状态包括以下所有数据（完整数据库副本）：

1. **视频列表**：所有已导入视频的文件路径、文件名、时长、分辨率、文件大小、修改时间
2. **视频状态**：`is_viewed`（已浏览）、`is_starred`（有收藏）、`is_exported`（已导出）
3. **分区信息**：每个视频的 A/B/C 等分区的起止时间、`is_viewed`（该分区已加载）、`has_starred`（分区内有收藏）、`has_exported`（分区内有导出）
4. **收藏记录**：收藏截图所属视频、分区标签、时间戳（毫秒）、截图路径（NAS路径）、是否已导出
5. **排除区间**：每个视频的全局排除区间（视频级别，所有分区共用）
6. **数据库版本**：数据库架构版本号

> **注意**：备份文件为完整的 SQLite 数据库文件（`.db`），不包含截图图片文件（.jpg）和缓存文件。


### 自动恢复备份功能（✅ 已开发完成）

每次启动程序时，系统自动执行以下流程：

1. 从配置中读取备份目录路径
2. 扫描备份目录，查找所有 `coverpicker_backup_*.db` 文件
3. 按修改时间排序，找到最新的备份文件
4. 获取当前数据库文件（`~/.coverpicker/coverpicker.db`）的修改时间
5. 如果最新备份文件的修改时间 > 当前数据库的修改时间，则自动恢复
6. 恢复后**无需重启**，界面直接使用恢复后的数据库
7. 如果恢复失败，静默跳过，用户无感知


### 与"数据库备份自动恢复"（v3.2 待开发）的区别

| 对比项       | **自动恢复备份**（已开发）                       | **数据库备份自动恢复**（v3.2 待开发）     |
| ------------ | ------------------------------------------------ | ----------------------------------------- |
| **触发时机** | 每次软件启动时                                   | 启动时检测到数据库损坏后                  |
| **触发条件** | 备份文件比当前数据库更新                         | `PRAGMA integrity_check` 检测到数据库损坏 |
| **恢复依据** | 文件修改时间                                     | 数据库完整性检查失败                      |
| **用户感知** | 静默执行，用户无感知                             | 恢复完成后提示用户，失败时显示错误        |
| **目的**     | 将数据库恢复到最新备份状态（用户主动备份的版本） | 在数据库损坏时自动修复，防止数据丢失      |

**简单说**：
- **"自动恢复备份"**：每次启动时，如果有更新的备份，自动用最新的备份覆盖当前数据库。
- **"数据库备份自动恢复"**（待开发）：仅在数据库损坏时触发，自动从最近的备份恢复，并提示用户。

**修复：细选窗口导出后视频列表图标不刷新**

- **根本原因**：`zoom_dialog.py` 的 `export_selected` 未触发主界面刷新
- **修复方案**：添加 `self.controller._notify_data_changed()`


### 2026-07-23 (v2.5.1)

**修复：排除区间架构重构（问题7：C区截图不显示）**

- **现象**：特定视频点击C区后跳回A区显示空白
- **根本原因**：排除区间被错误设计为分区级别
- **修复方案**：
  1. 排除区间从 `segments` 表迁移到 `videos` 表，改为视频级别（全局）
  2. `load_excluded_ranges_from_db` 从 `videos` 表读取，不再重置 `current_seg_index`

**关键教训**：
- 排除区间是"视频级别"的概念，不是"分区级别"
- 跨设备同步时，应根据数据的"作用域"决定存储位置
