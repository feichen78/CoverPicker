禁止擅自删改功能：任何时候，不允许未经用户授权，删除或修改coverpicker已有的功能、布局、按钮等。如果确实需要更改，必须取得用户确认。如果不确定用户原来的代码已实现什么功能，绝不允许擅自开始修改或生成简化的文件代码，务必向用户询问、索取原始代码。如有必要，可以增加调试日志，不需要请用户确认。每次调试，用户都会使用你最后提供的版本完整覆盖，不需要对此提出质疑，不需要就此向用户确认。

自查后输出：输出代码前必须进行三轮独立的审查：

第一轮（技术审查）：检查代码是否有语法错误、逻辑漏洞、变量名错误、缩进错误、导入缺失等技术性问题。

第二轮（需求审查）：以用户的视角模拟运行，在脑海中逐行执行代码，检查修改后的代码是否真正实现了用户描述的需求，而不是仅仅"能运行"。

第三轮（规则审查）：仔细核对前面3条规则，是否有违反任何一条。

三轮审查全部结束后，必须用中文输出一份简短的"需求实现确认清单"，至少包含，确认修改后的代码确实实现了用户的需求，且未违反核心规则。不允许输出"已自查3次，确认无误"这样空洞的结论，必须具体说明需求如何被实现。

## 🔴 已修复问题（v2.5.1）

### 问题7：C区截图不显示（排除区间全覆盖）—— ✅ 已修复

**现象**：特定视频在点击C区后，跳回A区并显示一片空白，但日志显示C区截图已成功生成。

**根本原因**：
1. 排除区间被错误地设计为**分区级别**（每个分区独立存储），而用户期望的是**视频级别（全局）**。
2. `load_excluded_ranges_from_db` 从 `segments` 表读取排除区间时，只读取当前 `current_seg_index` 对应分区的数据。
3. 当异常视频的所有分区都被写入了相同的排除区间时，切换到C区触发了 `_recalculate_segments_with_exclusions`，内部强制重置 `current_seg_index = 0` 并清空 `screenshots`。
4. 虽然 `_load_segment(2)` 成功生成了C区的截图，但 `current_seg_index` 已被重置为0，UI 刷新时始终读取 `screenshots["A"]`，而A区数据已被清空，导致空白。

**修复方案**：
1. **架构重构**：将排除区间从 `segments` 表迁移到 `videos` 表，改为视频级别（全局）存储。
2. 所有分区共用一套排除区间。
3. `load_excluded_ranges_from_db` 从 `videos` 表读取全局排除区间，不再重置 `current_seg_index`。
4. 升级后旧数据不迁移，用户需重新设置排除区间。
5. `segments` 表中的 `excluded_ranges` 列保留但不再使用。

**关键教训**：
- **排除区间是"视频级别"的概念，不是"分区级别"**。用户设置排除区间是为了排除片头片尾，这个设置应该对整个视频生效，而不是每个分区独立。
- 实现"跨设备持久化"时，我错误地理解了需求，将排除区间存储到了 `segments` 表，导致每个分区都有自己的排除区间。
- 正确的设计应该是：一个视频只有一套排除区间，所有分区共用。
- **跨设备同步的本质是同步整个数据库**，不代表需要把每个字段都复制到每个分区。应该根据数据本身的"作用域"来决定存储位置。
